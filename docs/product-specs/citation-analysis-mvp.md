# 引用分析智能体 MVP 产品规格

## 0. 原始需求

设计一个能够自主完成学术影响力全流程分析的智能体，主动爬取施引文献、自动识别重量级学者身份、分析引用语境与情感，并自主整合多维信息生成完整的可视化分析报告。智能体需具备主动信息获取、跨源数据融合与自主报告生成的能力。

- 文献爬取智能体：给定一篇目标论文，主动爬取所有施引文献的元数据信息。调用：`Semantic Scholar API`、`CrossRef API`、`Google Scholar` 爬虫工具
- 学者识别智能体：对施引作者进行 `h-index`、机构、领域权威度评估，标注重量级学者。调用：学者信息 API 工具（`OpenAlex / DBLP`）
- 引用情感分析智能体：提取每篇施引文献中提及目标论文的上下文句子，判断是正向引用 / 中性介绍 / 批评性引用。调用：全文爬取工具、情感分类工具（可用 `LLM zero-shot`）
- 可视化报告智能体：整合以上数据，生成引用趋势图、引用来源地图、引用情感分布图。调用：`Matplotlib / Pyecharts` 绘图工具、PDF 报告生成工具

## 1. 背景与目标

### 背景

当前针对一篇目标论文的学术影响力分析主要依赖人工在多个学术平台之间检索、比对和阅读。研究者通常能看到被引次数，但难以系统回答“哪些论文在引用”“哪些作者属于重量级学者”“引用语境是支持、介绍还是批评”“引用来源在时间和地域上如何分布”这类更高价值的问题。

### 目标

构建一个围绕单篇目标论文展开的学术影响力分析智能体系统，自动收集施引文献与作者信息，识别重量级学者，分析引用语境与情感，并生成一份面向研究判断的可视化分析报告。

### 为什么现在做

现有学术检索工具大多只提供基础检索与计数能力，缺少围绕“施引文献、学者影响力、引用语境、可视化报告”展开的一体化分析能力。随着公开学术元数据 API、全文获取工具和大模型分类能力逐步可用，做一个端到端的学术影响力分析闭环已经具备现实条件。

## 2. 目标用户

### 主要用户

科研人员、综述作者、审稿人、课题组负责人。

### 次要用户

科研管理者、学术情报分析人员、基金评审支持人员。

### 用户当前痛点

- 施引文献分散在多个学术平台，收集成本高。
- 仅靠引用次数难以判断目标论文的真实影响力结构。
- 难以快速识别哪些施引作者属于重点学者或高影响力作者。
- 难以快速区分支持性引用、背景性引用和批评性引用。
- 缺少可直接展示引用趋势、来源分布和情感分布的整合报告。

## 3. MVP 范围

### 包含

- 输入一篇目标论文的 DOI、标题、论文 ID 或 arXiv 链接。
- 主动抓取所有可获得的施引文献元数据。
- 获取施引作者、机构和基础学术影响力信息。
- 对施引作者进行重量级学者标注。
- 提取引用上下文并做基础情感分类。
- 生成单次分析的可视化报告。

### 不包含

- 多篇目标论文的联合分析结果输出。
- 面向整个研究主题的大规模持续监控。
- 自建高精度监督式情感分类模型训练。
- 面向机构级生产环境的权限、计费和多租户能力。
- 实时网页产品化交互系统。

### 第一版边界

第一版仅面向“单篇目标论文”的单次分析闭环，优先验证施引文献抓取、学者识别、引用情感分析和报告生成这四条主链路是否可跑通，不追求最全覆盖率和最强模型精度。架构上可为多论文扩展预留接口，但 MVP 不交付多论文联合分析能力。

## 4. 输入与输出

### 输入

- 目标论文 DOI。
- 目标论文标题。
- 目标论文在 `Semantic Scholar` / `OpenAlex` 等平台中的论文 ID。
- arXiv 链接。
- 可选分析参数：时间范围、数据源开关、报告输出格式。

### 输出

- 施引论文列表及其元数据。
- 作者、机构、领域影响力汇总结果。
- 重量级学者标注结果。
- 每条引用记录对应的引用上下文与情感标签。
- 可视化分析报告。

### 输入约束

- DOI、标题、论文 ID、arXiv 链接至少提供一种。
- 若仅提供标题且存在歧义，系统应要求或自动尝试补充校验信息。
- 若外部数据源无法确认目标论文唯一身份，系统应返回不确定状态而不是输出错误分析结果。

### 输出形式

- 结构化 JSON 结果。
- 单次分析报告，可优先支持 HTML，后续扩展 PDF。
- 图表资源文件，如 PNG / SVG / ECharts 配置。

补充定义：

- 引用来源地图基于施引作者所属机构的国家 / 地区信息生成，用于展示目标论文影响力的地理来源分布。若一篇施引论文存在多个作者机构，第一版优先按第一作者机构统计；若机构信息缺失，则该记录不纳入地图统计。

## 5. 核心能力

### 5.1 文献爬取智能体

#### 文献爬取目标

围绕目标论文主动收集所有可获得的施引文献信息，并尽量对齐不同平台之间的记录。

#### 文献爬取输入

- 目标论文 DOI / 标题 / 论文 ID / arXiv 链接。
- 外部数据源配置。

#### 文献爬取输出

- 施引论文清单。
- 每条施引记录的标题、作者、年份、来源、DOI、链接等元数据。
- 每条记录的来源标识与去重后的统一 ID。

#### 文献爬取数据源

- `Semantic Scholar API`
- `Crossref API`
- `Google Scholar` 爬取工具

补充说明：

- `Google Scholar` 作为补充源纳入 MVP 设计，但不作为主流程依赖。
- `arXiv` 仅作为目标论文输入兼容入口，不作为施引数据主抓取源。

### 5.2 学者识别智能体

#### 学者识别目标

对施引作者进行基础学术影响力评估，并标注重量级学者，为报告中的影响力判断提供辅助信息。

#### 学者识别输入

- 作者姓名。
- 作者所属论文与机构信息。
- 可关联的作者 ID。

#### 学者识别输出

- 作者的 `h-index`、机构、研究领域、代表性平台标识。
- 重量级学者标注结果。
- 作者与机构的聚合统计信息。

#### 学者识别数据源

- `OpenAlex`
- `DBLP`

#### 重量级学者标注规则

- 第一版采用启发式规则标注“重量级学者候选”，不做跨领域标准化排名。
- 若作者在可获取数据源中的 `h-index >= 30`，则标记为“高影响力作者候选”。
- 若作者在目标论文的施引文献集合中出现 2 次及以上，则提高其候选优先级。
- 若作者同时满足 `h-index` 达标和多次施引两个条件，则标记为“重量级学者候选”。
- 若缺少 `h-index` 数据，则仅基于施引频次和机构信息做弱标注，并在结果中注明“证据不足”。

### 5.3 引用情感分析智能体

#### 情感分析目标

提取施引文献中提及目标论文的上下文句子，并判断该引用是正向支持、中性介绍还是批评性质。

#### 情感分析输入

- 施引论文全文或可提取文本。
- 目标论文在施引论文中的引用标记。

#### 情感分析输出

- 引用上下文片段。
- 每条引用的情感标签。
- 可选置信度或解释文本。

#### 方法约束

- 第一版允许使用基于规则或 `LLM zero-shot` 的弱监督分类方法。
- 若无法提取全文或上下文，应明确标记为“无法判断”，而不是强行分类。
- 第一版不提供人工复核接口，直接输出模型或规则的分类结果。

### 5.4 可视化报告智能体

#### 报告生成目标

整合抓取、学者识别和情感分析结果，生成适合研究判断和快速阅读的可视化分析报告。

#### 报告生成输入

- 施引元数据。
- 作者与机构信息。
- 重量级学者标注结果。
- 情感分类结果。

#### 报告生成输出

- 报告摘要。
- 关键发现列表。
- 图表与结构化结果导出。

#### 报告内容

- 目标论文基础信息。
- 引用数量与年份趋势。
- 引用来源地图。
- 重量级学者分布与代表作者列表。
- 引用情感分布。
- 主要观察结论与待人工关注项。

## 6. 重量级学者标注定义

### 标注目标

重量级学者标注用于帮助用户快速识别在施引作者中具有较高学术影响力或领域代表性的作者，不作为正式学术评价结论。

### MVP 最小判定规则

- 第一版采用启发式规则标注“重量级学者候选”，不做跨领域标准化排名。
- 若作者在可获取数据源中的 `h-index >= 30`，则标记为“高影响力作者候选”。
- 若作者在目标论文的施引文献集合中出现 2 次及以上，则提高其候选优先级。
- 若作者同时满足 `h-index` 达标和多次施引两个条件，则标记为“重量级学者候选”。
- 若缺少 `h-index` 数据，则仅基于施引频次和机构信息做弱标注，并在结果中注明“证据不足”。

## 7. 验收标准

### 功能验收

- 系统可针对单篇目标论文完成一次端到端分析。
- 系统可输出施引论文清单和基础作者信息。
- 系统可对施引作者完成重量级学者标注。
- 系统可对部分可获取上下文的引用完成情感分类。
- 系统可生成包含图表和摘要的分析报告。

### 数据验收

- MVP 默认接入 `Semantic Scholar API`、`Crossref API` 和 `OpenAlex`。
- `Semantic Scholar + Crossref` 作为论文元数据主链路，`OpenAlex` 作为作者与机构补充源。
- `Google Scholar` 不作为 MVP 验收必需项，仅作为补充源。
- 每条去重后的施引记录必须至少保留：标题、作者、年份、来源、统一 ID。
- 每条记录必须能够追溯到至少一个原始数据源。

### 报告验收

- 报告至少包含引用趋势图、引用来源地图、引用情感分布图。
- 报告应包含重量级学者列表或重点作者分布。
- 报告应标记数据缺失和低置信度部分。

### 性能与时效

- 对施引论文数量不超过 100 篇的目标论文，单次完整分析应在 10 分钟内完成。
- 对只执行元数据抓取和学者识别的分析任务，应在 3 分钟内完成。
- 当全文缺失或外部 API 失败时，系统应输出降级报告，而不是整体失败。
- 降级报告必须明确标记未完成的数据源、未执行的模块和可能影响的结论。

## 8. 非功能要求

### 稳定性

外部数据源部分失败时，系统仍应输出部分结果和失败说明。

### 可解释性

每一项重量级学者标注和情感分类结果应尽量附带来源、上下文或解释文本，便于人工理解。

### 可扩展性

系统应允许后续新增更多数据源、替换情感分类器和扩展报告模板。

### 合规与伦理

系统应遵守外部数据源使用条款，不将自动标注结果包装为绝对事实，应强调其为辅助判断工具。

## 9. 外部依赖与风险

### 外部 API / 数据源

- `Semantic Scholar API`
- `Crossref API`
- `OpenAlex / DBLP`
- `Google Scholar` 爬取工具
- `arXiv API`
- 全文抓取或解析工具

### 技术风险

- 外部 API 限流或字段不一致。
- 全文难以稳定获取。
- 引用上下文定位存在误差。
- `LLM zero-shot` 情感分类稳定性有限。
- 重量级学者标注规则可能受数据源完整性影响。

### 数据风险

- 不同数据源之间存在记录缺失、重复或冲突。
- 作者消歧与机构归属可能不准确。
- 某些论文没有开放获取全文，导致情感分析覆盖不完整。
- 不同平台的作者影响力指标口径可能不一致。

### 运营风险

- `Google Scholar` 爬取存在稳定性和合规风险。
- 外部依赖变化可能导致结果波动。
- 报告若被过度解读，可能带来错误学术判断风险。

## 10. 待确认问题

- `Google Scholar` 作为补充源纳入 MVP 设计，但不作为主流程依赖。
- 重量级学者标注是否需要为不同领域设置不同阈值？无需
- PDF 是否作为 MVP 必交付格式，还是 HTML 报告优先？第一版先完成HTML
