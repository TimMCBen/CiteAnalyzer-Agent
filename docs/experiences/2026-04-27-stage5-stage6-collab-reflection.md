## RALPLAN 反思计划

### Principles

1. **先冻结阶段边界，再改实现细节。**
2. **默认主路径和兼容增强路径要明确区分。**
3. **最新用户指令优先于上一轮局部设计。**
4. **代码契约变化必须同轮同步文档。**
5. **验证结果要能区分本地夹具、真实样本和在线依赖。**

### Decision Drivers

1. 今天的核心冲突不是“能不能实现”，而是 **stage5/stage6 的默认契约到底是什么**。
2. 多轮对话里用户多次切换过偏好：从 `TeX` 精定位、到 `PDF-first + tar sidecar`、再到最后明确选择 **方案 B / PDF-first**。
3. 文档一旦不跟随最终契约收口，就会让 README、测试说明、执行计划与代码同时漂移。

### Viable Options

#### Option A：把这轮经验直接写进长期规范

##### Pros

- 后续最容易被看到

##### Cons

- 这轮经验还没有跨多个任务稳定复用
- 会把局部协作经验直接升级为默认规则，风险过高

#### Option B：先写入经验池，再由人决定是否升级

##### Pros

- 符合 `docs/EXPERIENCE_GOVERNANCE.md` 的机制
- 能保留过程经验，又不污染仓库长期基线
- 适合记录“今天这种多轮收口 + 文档同步”的方法论

##### Cons

- 默认可见性不如长期规范高

#### Option C：只写 history，不做经验沉淀

##### Pros

- 改动最小

##### Cons

- history 更偏“做了什么”，不够承载“以后怎么协作更稳”

### Recommendation

选择 **Option B**：先写入经验池。

原因是这轮任务已经形成了可复用的协作经验，但还不到应该自动升格为仓库长期规则的程度。

## 今日对话中提炼出的可复用经验

### EXP-2026-04-27-01：复杂阶段重构要先冻结“阶段契约”，再改代码

- **分类**：`任务拆分`
- **问 / 发生了什么**：
  今天真正反复变化的不是某个函数实现，而是 `stage5` 和 `stage6` 的职责边界与默认主路径。
- **答 / 经验是什么**：
  当任务涉及多个阶段、多个消费方、多个文档时，应先用短版 `ralplan` 冻结：
  - 阶段职责
  - 默认主路径
  - 兼容增强路径
  - 验证口径
  然后再动实现。
- **适用场景**：
  - 多阶段 pipeline 重构
  - 上下游契约联动修改
  - 既要改代码又要改文档的任务
- **反例 / 失败模式**：
  如果先写代码、后补契约，很容易出现：
  - README 说的是 A
  - 测试验证的是 B
  - 执行计划还停留在 C
- **建议升级**：`建议升级`

### EXP-2026-04-27-02：默认主路径和兼容路径必须分开表述

- **分类**：`文档同步`
- **问 / 发生了什么**：
  今天对 `PDF`、`TeX`、`tar sidecar` 的讨论反复切换，如果不明确“哪个是默认主路径、哪个只是兼容增强”，代码和文档就会不断打架。
- **答 / 经验是什么**：
  对这种多路径系统，文档里要显式写成：
  - 默认主路径是什么
  - 退化 / 兼容路径是什么
  - 哪条路径是“应该被优先验证的”
- **适用场景**：
  - PDF vs TeX
  - API 主链路 vs fallback
  - 在线依赖 vs 本地夹具
- **反例 / 失败模式**：
  不分层表述会让团队成员误以为所有路径地位相同，最终导致验证目标发散。
- **建议升级**：`建议升级`

### EXP-2026-04-27-03：最新用户指令应覆盖前一轮局部设计，但要在结果里显式说明覆盖关系

- **分类**：`多智能体协作`
- **问 / 发生了什么**：
  用户先要求 `PDF-first + tar sidecar`，随后又明确选择 **方案 B**，要求不再保留 tar 作为默认正式产物。
- **答 / 经验是什么**：
  当用户后续指令与上一轮局部设计冲突时，应：
  1. 采用最新指令
  2. 在执行前明确告诉用户“这会覆盖上一轮哪条局部设计”
  3. 在提交说明和文档里写明这种覆盖关系
- **适用场景**：
  - 同一任务中途换方案
  - 用户先探索，再拍板
- **反例 / 失败模式**：
  如果只默默切换，不写明覆盖关系，后续很难解释为什么代码和早先讨论不一致。
- **建议升级**：`仅记录`

### EXP-2026-04-27-04：代码提交和文档提交分两笔时，要把第二笔当作“契约收口”，不是补丁

- **分类**：`Git流程`
- **问 / 发生了什么**：
  今天用户明确要求“先提交一版，再整理所有 doc 文件后再提交”。
- **答 / 经验是什么**：
  这种两笔提交不是“代码 + 零散文档”，而应该是：
  - 第一笔：实现 + 测试
  - 第二笔：契约收口 + 经验留痕 + 文档同步
- **适用场景**：
  - 用户希望把实现和文档拆开审阅
  - 一个分支里有明显的“行为变更”和“说明收口”两个阶段
- **反例 / 失败模式**：
  如果第二笔只做零碎补文档，后续 PR 阅读者会看不清哪一笔才真正完成契约收口。
- **建议升级**：`建议升级`

### EXP-2026-04-27-05：验证输出要明确区分“本地夹具验证”和“真实在线验证”

- **分类**：`测试策略`
- **问 / 发生了什么**：
  今天用户多次追问“到底抓到的是真实全文，还是测试夹具”“本地是否能看到原文”。
- **答 / 经验是什么**：
  测试报告里应该显式区分：
  - 本地夹具验证
  - 真实样本验证
  - 在线依赖验证（例如 GROBID / arXiv）
- **适用场景**：
  - 既有 mock/fixture，也有 live smoke 的项目
  - 用户关心是否真的命中了外部资源
- **反例 / 失败模式**：
  如果只说“测试通过”，用户很容易误解为所有通过都基于真实数据。
- **建议升级**：`建议升级`

### EXP-2026-04-27-06：PR 操作前先查 PR 状态，避免在已合并 PR 上继续补描述

- **分类**：`PR协作`
- **问 / 发生了什么**：
  今天推送并开 PR 时，发现旧的 `#21` 已经是 `MERGED`，因此重新开了 `#22`。
- **答 / 经验是什么**：
  在同一分支继续推进前，先做一次：
  - `gh pr view`
  - 确认 PR 是否已存在
  - 确认是 `OPEN`、`MERGED` 还是 `CLOSED`
  再决定是更新旧 PR，还是新开一个。
- **适用场景**：
  - 长生命周期分支
  - 同一主题多次迭代
- **反例 / 失败模式**：
  如果默认认为 PR 还开着，可能会把新的提交错误地挂到已合并上下文里。
- **建议升级**：`仅记录`

## 对今天这轮协作的总体反思

### 做得对的地方

- 最后能够把 `stage5/stage6` 的默认契约收口到一套一致说法
- 两次提交的拆分清楚，便于审阅
- PR、quickscore、history、README、执行计划、测试说明最终都同步到了同一方向

### 做得不够好的地方

- 中间一度把 `TeX` 精定位偏好和 `PDF-first` 默认主路径混在一起，导致说明反复摇摆
- 在用户只发 `stage` 这种短消息时，没有足够快地把上下文收束成单一下一步
- 对“本地夹具验证”和“真实抓取验证”的区别，最初解释得不够早

### 后续如果再做类似任务，建议默认采用的节奏

1. 先用短版 `ralplan` 冻结阶段契约
2. 明确默认主路径 / 兼容路径
3. 第一笔提交只放实现与测试
4. 第二笔提交专门做文档收口与 history / experience 记录
5. PR 前先检查旧 PR 状态
