# 协作经验沉淀机制

这份文档定义一套**项目无关、可复用**的机制，用于在 Agent 协作开发过程中沉淀经验，同时控制哪些经验可以自动记录、哪些经验需要人工确认后才能升级成正式规则。

它适用于：

- Agent 与人协作开发代码项目
- 多轮任务推进
- PR / review / CI / 测试 / API 调试等协作过程

它**不依赖某个具体业务项目**，而是面向协作开发过程本身。

## 目标

这套机制想解决两个相反的问题：

1. 有价值的协作经验如果只留在聊天里，后续很容易丢失。
2. 如果把每一条局部经验都直接写进正式规范，又会污染仓库的初始基线文件。

因此机制的核心原则是：

**经验可以自动入池，规则升级必须人工确认。**

## 两层结构

### 1. 经验池

经验池用于存放：

- 当轮任务中形成的可复用经验
- 尚未升格为正式规范的协作方法
- 局部有效、但值得保留的实践结论

经验池文件可以由 Agent 自动追加或整理。  
它的定位是：

- 候选知识
- 工作经验
- 复盘材料

而不是仓库长期规范本身。

### 2. 正式规范

正式规范用于存放：

- 默认入口
- 默认协作方式
- 默认测试 / CI 约束
- 长期有效的团队规则

正式规范不能因为某一轮对话中的局部经验就自动修改。

## Agent 可以自动做什么

Agent 可以自动：

- 判断本轮是否形成了可跨项目复用的协作经验
- 将经验写入经验池
- 给经验添加类别、适用场景和建议升级标记
- 在收尾时提醒是否有经验值得人工审阅

Agent 不应该自动：

- 把经验直接升格成团队默认规则
- 因为一轮任务的局部经验就修改长期规范文件

## 必须人工确认的情况

如果某条经验可能影响以下内容，就必须由人决定是否升级：

- 仓库初始基线文件
- 默认入口文件
- 模板文件
- 长期规范文件

判断标准优先看**文件职责**，而不是只看文件名。

### 典型高谨慎文件

以下类型文件通常属于高谨慎范围：

- 仓库入口文件
  - 如 `README.md`
  - 如 `AGENTS.md`
- 协作与规则文件
  - 如 `CONTRIBUTING.md`
  - 如仓库级 guide / policy / governance 文档
- 工程基线文件
  - 如 `Makefile`
  - 如 `.gitignore`
  - 如 `.editorconfig`
  - 如 `CODEOWNERS`
- 模板级流水线与安全文件
  - 如 `.github/workflows/*`
  - 如 `docs/CICD.md`
  - 如 `docs/SECURITY.md`
  - 如 `docs/SUPPLY_CHAIN_SECURITY.md`

如果经验会改变这些文件所表达的默认规则，就必须人工确认。

## 什么经验适合先入池

优先记录这几类：

- PR / review 协作经验
- 测试与验证策略
- API 调试与额度控制经验
- 多智能体协作边界
- Git / 分支 / stacked PR 经验
- 文档同步和知识沉淀经验

不建议记录的内容：

- 只在这一次场景下成立的偶然现象
- 纯临时状态
- 没有复用价值的流水账

## 升级成正式规范的判断标准

只有满足下面任一条件，才建议从经验池升级到正式文档：

- 已经在多个任务中重复验证有效
- 能明显减少重复错误或协作摩擦
- 会影响 PR、review、测试、分支、CI 等共通流程
- 已经不是“这次特例”，而是稳定适用的协作做法

即使满足这些条件，也仍然需要人工确认。

## 推荐工作流

### 每轮收尾

1. Agent 判断是否形成了可复用经验
2. 若有，则写入经验池
3. 给出标记：
   - `仅记录`
   - `建议升级`
4. 若可能影响初始基线文件或长期规范文件，则明确提醒需要人工判断

### 人工筛选

1. 人判断经验是否足够稳定
2. 人决定是否升级到正式文档
3. 若升级，则由 Agent 协助修改正式规范文件并同步相关文档

## 推荐记录格式

经验池中推荐每条经验至少包含：

- 标题
- 问：发生了什么问题 / 什么时候适用
- 答：经验是什么
- 适用场景
- 反例或失败模式
- 是否建议升级

## 推荐分类

建议按以下类别标记：

- `任务拆分`
- `PR协作`
- `Review处理`
- `测试策略`
- `API调试`
- `多智能体协作`
- `文档同步`
- `Git流程`

## 一句话原则

> Agent 可以自动记经验，但不应该自动立规矩。
