# 质量评分

用这份文档按产品区域和架构层次记录当前质量水位，方便持续知道最薄弱的地方在哪。

## 建议的评分标准

- `A`：覆盖完整、行为稳定、文档清楚、运行风险低。
- `B`：整体可接受，但还有明确短板。
- `C`：能用，但需要针对性补强。
- `D`：脆弱、缺少规范，或很多行为尚未定义。

## 初始模板

| 区域 | 评分 | 原因 | 下一步 |
| --- | --- | --- | --- |
| 产品面 | D | 还没有真实产品定义。 | 先明确第一个用户路径和验收标准。 |
| 架构文档 | B | 总体边界、阶段 1 / 阶段 2 结构和执行计划已经比较清楚，阶段 5 / 6 的 PDF-first 与 GROBID 主路径也已补充，但阶段 4 和最终报告链路仍未完全细化。 | 继续同步阶段 4 与阶段 7 的输入输出边界，并把总智能体联调路径补齐。 |
| 测试 | B | 已有阶段 1 / 阶段 2 脚本验证，阶段 5 / 6 也已经有可运行的本地验证入口，但真实样本回归仍然偏少。 | 为阶段 5 / 6 增加更多真实 PDF 样本回归，并继续补全端到端验证。 |
| 可观测性 | C | 已有 `FetchSummary`、来源追踪和基础错误暴露，但请求日志、rate limit 观测仍然不足。 | 补充外部 API 请求日志、重试信息和失败分类。 |
| 安全 | C | 默认约束已经有了，但具体实现还没落地。 | 根据项目接入真实认证、密钥和依赖治理。 |
