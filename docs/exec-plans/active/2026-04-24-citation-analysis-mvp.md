# CiteAnalyzer-Agent MVP 执行计划

## 目标

完成 `CiteAnalyzer-Agent` 的 MVP 实施计划，并按“论文被引分析智能体 + 子智能体”架构推进首轮开发。首轮目标是围绕单篇目标论文，跑通施引文献抓取、学者识别、引用情感分析和 HTML 报告生成的端到端闭环。

## 范围

- 包含：
  - 单篇目标论文输入解析与统一标识
  - 施引文献抓取与多源融合
  - 学者画像补充与重量级学者标注
  - 引用上下文提取与情感分类
  - HTML 报告生成
  - 端到端样本验证
- 不包含：
  - 多篇目标论文联合分析
  - 主题级持续监控
  - PDF 首版交付
  - 实时 Web 产品化界面
  - 自训练高精度情感模型

## 背景

- 相关文档：
  - `docs/product-specs/citation-analysis-mvp.md`
  - `docs/ARCHITECTURE.md`
- 相关代码路径：
  - 当前仓库尚未建立真实实现目录，计划逐步落地到 `apps/analyzer/` 与 `packages/*`
- 已知约束：
  - `Semantic Scholar + Crossref` 为主抓取链路
  - `Google Scholar` 作为补充源，不阻塞主流程
  - `arXiv` 仅作为输入兼容入口
  - HTML 为当前默认报告交付格式
  - 情感分析仅对成功提取上下文的记录执行
  - 重量级学者标注采用启发式规则，不做跨领域标准化排名

## 风险

- 风险：外部 API 限流或字段不一致
  - 缓解方式：优先实现主链路，保留来源追踪和降级逻辑
- 风险：全文难以稳定获取，影响情感分析覆盖率
  - 缓解方式：允许输出“无法判断”，不阻塞报告生成
- 风险：作者消歧与机构归属不准确
  - 缓解方式：首版使用弱标注，并在结果中保留“证据不足”
- 风险：双人并行开发导致接口反复变化
  - 缓解方式：由主导人统一维护共享数据结构和模块接口
- 风险：报告生成阶段发现上游字段不统一
  - 缓解方式：在各阶段先定义标准化输出，再做集成

## 决策记录

- 2026-04-24：系统采用“一个总智能体 + 多个子智能体”的架构，以保持产品表达和实现边界一致。
- 2026-04-24：`Semantic Scholar + Crossref` 作为主抓取链路，`Google Scholar` 仅作为补充源。
- 2026-04-24：`arXiv` 仅作为输入兼容入口，不作为施引抓取主来源。
- 2026-04-24：首版报告以 HTML 为默认交付格式。
- 2026-04-24：重量级学者采用启发式规则标注，不做跨领域标准化排名。
- 2026-04-24：引用来源地图以施引作者所属机构的国家 / 地区信息为准，多机构时优先按第一作者机构统计。
