# 参与协作

这个仓库是为 Agent-first 开发准备的，但这些规则对人和 Agent 都一样适用。

## 基本协作方式

- 从 `AGENTS.md` 开始，再按任务类型去读对应文档。
- 仓库级知识要落在版本化文件里，不要只存在聊天记录、口头同步或工单评论里。
- 如果行为变了，就一起更新代码、文档、测试和 release/history 记录。
- 遇到跨度大、风险高、会分多轮推进的任务，先在 `docs/exec-plans/active/` 下建 execution plan。

## 发起 Pull Request 之前

- 运行 `make ci`。
- 如果本次改动涉及代码或仓库流程，补齐或更新对应 history。
- 如果变更对用户可感知，补齐 release note。
- 确认示例、脚本、说明文档和当前实现一致。

## 使用本地 CLI 创建远程 PR 时的填写与兜底要求

- 当用户明确要求 Agent 创建远程 PR 时，Agent 应先尝试使用本地可用的 CLI 工具完成，而不是默认改走其他远端方式。
- 如果本地 CLI 不存在、不可用或无法完成创建，Agent 应明确告知用户当前缺少可用 CLI，并让用户改为手动创建 PR。
- 当本地 CLI 为 `gh` 时，Agent 应主动根据当前分支的真实变更填写 PR 标题和正文，而不是只粘贴空模板。
- PR 标题要明确反映本次变更范围，不要直接使用分支名、模糊概括或占位标题。
- PR 正文默认基于 `.github/pull_request_template.md` 填写完整内容。
- 如果当前分支相对 base 分支包含超出本轮任务范围的历史提交，Agent 应先提示范围问题，再决定是继续创建 PR、拆分分支，还是按真实 diff 重写摘要。

## Review 默认要求

- 优先拆成范围清晰的小 PR。
- 明确写出风险点、迁移影响和后续待办。
- 如果上下文复杂，直接链接对应 plan、spec 或 history，不要依赖评审者自己猜。
